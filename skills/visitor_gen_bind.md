# Visitor & gen_bind — Adding or Modifying Visitor Methods

This skill covers modifying the AST visitor (`Visitor` class) — adding new virtual methods, changing signatures, or adding new `canVisit*` gates. The visitor is reflected into daScript via `gen_bind.das`, which generates C++ adapter code. This process spans **7 files** and requires a specific build sequence.

## Build Sequence (ORDER MATTERS)

```
1. Edit C++ headers + source (visitor, adapters, ast_infer_type)
2. Edit src/builtin/ast.das (daScript-side visitor class)
3. Build → CMake regenerates ast.das.inc (embedded byte array of ast.das)
4. Run gen_bind → regenerates ast_gen.inc from the NEW binary
5. Update adapter implementations to use generated get_/invoke_ helpers
6. Rebuild → final binary with all changes
```

**Critical**: gen_bind runs using `daslang.exe`, which embeds `ast.das` as a compiled-in byte array (`ast.das.inc`). If you run gen_bind before building, the binary still has the OLD `ast.das` and gen_bind won't see your new methods. You MUST build first (step 3) so the binary has the updated `ast.das.inc`, then run gen_bind (step 4).

## Files to Modify

### 1. `include/daScript/ast/ast_visitor.h` — C++ virtual methods

Add the new virtual method to the `Visitor` class:

```cpp
virtual bool canVisitMyNewThing ( ExprMyThing * /*expr*/ ) { return true; }
```

### 2. `src/ast/ast.cpp` — Gate the expression visit

If adding a `canVisit*` gate, wrap the expression's `visit()` method:

```cpp
ExpressionPtr ExprMyThing::visit(Visitor & vis) {
    if ( vis.canVisitMyNewThing(this) ) {
        // ... existing body unchanged ...
    } else {
        return this;
    }
}
```

### 3. `src/builtin/ast.das` — daScript abstract method

Add matching abstract method to the `AstVisitor` class. Position matters — it determines the field index in the generated adapter.

```das
def abstract canVisitMyNewThing(expr : ExprMyThing?) : bool
```

**Type mapping**: raw pointer `ExprMyThing?` in daScript maps to `ExprMyThing *` in C++. Smart pointer `smart_ptr<ExprMyThing>` maps to `smart_ptr_raw<ExprMyThing>`. Match the pattern of existing methods (e.g. `canVisitCall` uses raw pointer, `preVisitExprCall` uses smart_ptr).

### 4. `include/daScript/simulate/aot_builtin_ast.h` — Adapter declaration

Add override declaration to `VisitorAdapter`:

```cpp
virtual bool canVisitMyNewThing ( ExprMyThing * expr ) override;
```

### 5. `src/builtin/module_builtin_ast_adapters.cpp` — Adapter implementation

**Before gen_bind**: add a placeholder so the build succeeds:

```cpp
bool VisitorAdapter::canVisitMyNewThing(ExprMyThing * /*expr*/) {
    return true; // placeholder until gen_bind regenerates
}
```

**After gen_bind**: replace with the real implementation using the generated helpers:

```cpp
bool VisitorAdapter::canVisitMyNewThing(ExprMyThing *expr) {
    if ( auto fnCanVisit = get_canVisitMyNewThing(classPtr) ) {
        bool result = true;
        runMacroFunction(context, "canVisitMyNewThing", [&]() {
            result = invoke_canVisitMyNewThing(context,fnCanVisit,classPtr,expr);
        });
        return result;
    } else {
        return true;
    }
}
```

The `get_*` and `invoke_*` helpers come from `ast_gen.inc` (generated in step 4).

### 6. `include/daScript/builtin/ast_gen.inc` — GENERATED, do not edit

Regenerated by running:

```bash
bin/Release/daslang_static.exe dasgen/gen_bind.das
```

This reads the `AstVisitor` class definition from the embedded `ast.das` and generates:
- Enum entries (`__fn_canVisitMyNewThing`)
- Field offset table entries
- `get_canVisitMyNewThing()` / `invoke_canVisitMyNewThing()` inline methods

Also regenerates `include/daScript/builtin/debugapi_gen.inc` (for debug walker adapters).

### 7. Additional files (if adding policies or serialization)

- `include/daScript/ast/ast.h` — add `CodeOfPolicies` fields (mark with `/*option*/` for auto-parsing)
- `src/builtin/module_builtin_rtti.cpp` — register new policy fields: `addField<DAS_BIND_MANAGED_FIELD(my_field)>("my_field")`
- `src/builtin/module_builtin_ast_serialize.cpp` — add to serialization chain: `<< value.my_field`

## Troubleshooting

### gen_bind produces no output for new methods

**Cause**: The binary used to run gen_bind still has the old `ast.das` embedded.

**Fix**: Build first (`cmake --build build --config Release`), then run gen_bind with the new binary.

**Verify**: `ast.das.inc` should be newer than `ast.das`. If not, touch `ast.das` and rebuild. The CMake `CMAKE_TEXT_XXD` macro compares file content (not timestamps) — if the hex representation matches, it skips regeneration.

### Link errors referencing `get_canVisitXxx`

**Cause**: Adapter implementation references generated helpers that don't exist yet.

**Fix**: Use placeholder implementations (just `return true;`) for the first build, then replace after gen_bind.

### Expression type not available as raw pointer in ast.das

All `Expr*` types used in visitor methods must be registered in the daScript module system. Check existing patterns — `ExprCall?`, `ExprNamedCall?`, `ExprLooksLikeCall?` all work as raw pointer types in `ast.das`. If a new expression type isn't registered, the abstract method will compile but gen_bind will silently skip it.

## Reference: Existing canVisit Gates

| Gate | Expression | Controls |
|---|---|---|
| `canVisitCall` | `ExprCall` | Call expression (resolved function call) |
| `canVisitNamedCall` | `ExprNamedCall` | Named call (before resolution to ExprCall) |
| `canVisitLooksLikeCall` | `ExprLooksLikeCall` | Base class for call-like expressions (covers ExprInvoke, ExprCallMacro, etc. but NOT ExprCall which overrides visit()) |
| `canVisitExprLooksLikeCallArgument` | argument of `ExprLooksLikeCall` | Individual argument visitation |
| `canVisitFunction` | `Function` | Function definition traversal |
| `canVisitStructure` | `Structure` | Structure definition traversal |
| `canVisitEnumeration` | `Enumeration` | Enumeration definition traversal |
| `canVisitGlobalVariable` | `Variable` | Global variable traversal |
| `canVisitMakeBlockBody` | `ExprMakeBlock` | Make-block body traversal |

## Expression Class Hierarchy (visitor-relevant)

```
Expression
├── ExprLooksLikeCall          ← canVisitLooksLikeCall gate
│   ├── ExprCallMacro          ← uses ExprLooksLikeCall::visit
│   ├── ExprCallFunc
│   │   ├── ExprCall           ← OWN visit() with canVisitCall gate
│   │   └── ExprNew
│   ├── ExprLikeCall<T>
│   │   └── ExprInvoke         ← uses ExprLooksLikeCall::visit
│   └── ...
└── ExprNamedCall              ← canVisitNamedCall gate (separate hierarchy)
```

`ExprCall::visit()` does NOT go through `ExprLooksLikeCall::visit()` — it has its own implementation. So `canVisitLooksLikeCall` does NOT affect `ExprCall` nodes (they use `canVisitCall` instead).
